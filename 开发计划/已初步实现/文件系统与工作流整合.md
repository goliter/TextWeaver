# 文件系统与工作流整合开发计划

## 一、开发目标

将虚拟文件系统正式接入工作流系统，实现以下功能：

1. **后端整合**：
   - 文件与工作流绑定，每个工作流拥有独立的文件系统
   - 新工作流创建时自动生成根目录
   - 数据库表结构更新，文件表添加flow_id外键

2. **前端整合**：
   - 实现三栏设计（左侧文件系统、中间工作流画布、右侧属性面板）
   - 文件系统显示当前工作流的文件结构
   - 文件内容兼容到右侧属性面板
   - 支持文件与工作流节点的交互

## 三、技术实现细节

### 3.1 后端实现

#### 数据库模型更新

- **文件表**：添加`flow_id`外键，关联到工作流表
- **工作流表**：添加与文件的一对多关系
- **级联删除**：工作流删除时自动删除相关文件

#### API路由更新

- 文件API路径更新为`/api/flows/{flow_id}/files`
- 所有文件操作都需要验证工作流的所有权
- 工作流创建时自动创建根目录文件

#### 工作流执行引擎

- 文件读取节点：通过`flow_id`和`file_id`读取文件内容
- 文件写入节点：通过`flow_id`和`file_id`写入文件内容
- 确保文件操作的权限验证

### 3.2 前端实现

#### 三栏布局

- 使用Tailwind CSS实现响应式三栏布局
- 左侧文件系统：固定宽度，可折叠
- 中间工作流画布：自适应宽度
- 右侧属性面板：固定宽度，根据选中内容动态切换

#### 文件系统集成

- 使用现有的文件树组件，修改API调用
- 显示当前工作流的文件结构，不显示根目录
- 支持文件的创建、编辑、删除操作
- 支持文件拖拽到工作流画布，自动创建文件节点

#### 右侧属性面板

- 选中文件时显示文件内容编辑器
- 选中节点时显示节点配置
- 支持文件内容的实时编辑和保存
- 集成Monaco Editor用于文件内容编辑

## 六、风险评估

| 风险                       | 影响                | 应对措施                     |
| -------------------------- | ------------------- | ---------------------------- |
| 数据库迁移失败             | 系统无法正常运行    | 提前备份数据库，准备回滚方案 |
| API路由变更影响现有功能    | 前端无法正常调用API | 渐进式更新，确保向后兼容     |
| 前端布局调整影响用户体验   | 用户操作不便        | 充分测试，确保布局合理       |
| 文件与工作流绑定后性能问题 | 系统响应缓慢        | 优化数据库查询，添加适当索引 |

## 七、验收标准

1. **功能验收**：
   - 新工作流创建时自动生成根目录
   - 文件系统显示当前工作流的文件结构
   - 支持文件的创建、编辑、删除操作
   - 支持文件拖拽到工作流画布
   - 右侧面板显示文件内容或节点配置
   - 工作流执行时能正确读写文件

2. **性能验收**：
   - 文件操作响应时间<1秒
   - 工作流执行时间符合预期
   - 页面加载时间<3秒

3. **用户体验验收**：
   - 三栏布局清晰合理
   - 文件系统操作直观易用
   - 工作流编辑流畅
   - 错误提示友好

## 八、总结

通过本开发计划，我们将实现虚拟文件系统与工作流系统的深度整合，为用户提供更加完整和高效的AI文本工作流解决方案。整合后的系统将支持文件与工作流的紧密关联，实现更加直观的文件管理和工作流编辑体验。
